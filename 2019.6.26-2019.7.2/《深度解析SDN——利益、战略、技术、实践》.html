第3章 SDN相关组织及利益纷争

ONF：Open Networking Foundation，非营利性组织，有Google、Facebook和微软等网络用户协同发起，成立于2011年。主要成果OpenFlow1.1-1.5协议标准、OF-Config1.0-1.1。ONF Plugfest进行Controller和OpenFlow Switch之间的协议测试和各个厂商的互联互通，每年两期，从2012年开始。
ONS：Open Networking Summit，负责组织开放网络研讨峰会的非营利性组织，由SDN的发起人创建，ONS有点像ONF的独立运行的全资子公司。
ODL：OpenDayLight，由设备商和软件商发起 ，于2013年创建。实现一个开源的SDN Controller。
NFV：Network Function Virtualization，由运营商于2012年提出。
IETF：由Cisco、Juniper主导，成立I2RS(Interface to Routing System)，将与配置有关的东西放到Controller上，具体的协议计算仍在每台网络设备上，Controller通过该接口控制每台设备的路由协议。

第4章 详解OpenFlow

详见OpenFlow协议
flow：在一段时间内，经过同一个网络的一系列具有相同属性的顺序发送的报文集合。相同属性在不同的应用情境下不同。
flow entry：
Match Fields Priority Counter Instruction Timeout Cookie

Instruction:选择其中的部分来介绍

  • Meter：用来测量该Flow的速率并执行响应的动作。每一个Meter包含几个Band，每个Band对应一个Rate和动作。Band是指如果所测量的Flow的速率超过了指定的Rate，就执行对应的动作。动作可以是drop/dscp remark(改写IP头部的dscp值)。

Action List：一个Instruction可能会指定对一个报文执行多个Action，这些Action均放在一个Action List中。
port：

  • 物理端口：交换机面板熵所有对外可见的物理端口；
  • 逻辑端口：一个例子就是Tunnel；
  • 保留端口：Controller Channel就是保留端口的一种。

Group：Action之一就是将Flow转发到一个Group。Bucket等同于一个Action List，但是仅在Group中使用。

  • All：转发到该Group所包含的所有buckets中。在芯片中以组播方式实现。
  • Select：转发到该Group所包含的所有Buckets中的一个。在芯片中以等级路由(ECMP)/端口聚合实现。
  • Indirect：多个Flow之间共享Action。这种类型的Group中只包含一个Bucke。，当多个Flow关联到同一个Indirect类型的Group时，芯片里就多个flow entry指向同一个下一跳表项，执行该Action。
  • Fail-over：正常状态下，报文只从这个Group的一条路径发出去，此路径断了，备份路径马上顶上。


在现有芯片上实现OpenFLow：

  • NPU(网络处理器)和FPGA(现场可编程们这列)方案：使用芯片的可编程代替交换机的可编程。
  • TTP(Table Typing Patterns)/NDM(Negotiabable Data-plane Model)方案：使用现有的芯片，组合成OpneFlow相应的功能。比如使用ACL表、路由表、Mac表、vlan表、port表、MPLS表等实现多级流表。

第5章 网络虚拟化和SDN


Hypervisor，虚拟化产品的核心，可以认为是虚拟机平台的超级管理系统。他可以访问服务器上包括磁盘、内存在内的所有物理设备。Hypervisor不但协调硬件资源的访问，也同时在各虚拟机之间施加防护。当服务器启动Hypervisor时，会加载所有虚拟机客户端的操作系统，同时为虚拟机分配适量的内存、CPU、网络和磁盘。Hypervisor内置虚拟交换机，支持相同服务器上的不同虚拟机之间的通信。所有交换机的虚拟网卡连接到虚拟交换机上，形成一个内部虚拟网络。


网络虚拟化：为了满足不同物理服务器上虚拟机的通信问题，尽可能将虚拟机的增删迁移和物理设备分离。虚拟机和虚拟机之间用Tunnel连接，Tunnel运行在物理网络之上，即overlay。Tunnel终结点放在服务器上，而不是边缘交换机上，在虚拟机迁移时，只要修改Tunnel的终结点关联即可。Tunnel携带一个独一无二的标志，表示独立的虚拟网络，具有天然的隔离性。

2008年，Cisco+VMware->Nexus 1000V，VxLan(Tunnel overlay)
2012年，Nicira->NVP(虚拟交换平台)


3种Tunnel技术的比较：

  • VxLan：Virtual Extensible LAN(虚拟可扩展局域网)，有Cisco、VMware、Arista等公司共同推出。将原始报文封装在一个UDP Tunnel Header里(z在原始报文前加入L2+IP+UDP+VxLan头)，并通过VxLan头里面的VNI信息将传统网络中的Vlan从4KB扩展到16MB，即一物理网络可以创建最多16MB个虚拟网络。
  • NvGRE：Network Virtualization using Generic Routing Encapsulation，微软推出。通过复用GRE技术，重新解释GRE Tunnel头中的GRE Key字段，将之高24bit用于VNI ID，从而实现扩展。
  • STT：Stateless Transport Tunning，Nicira推出，只能用于NVP上，解决对CPU性能影响较大的报文分片问题。该技术的Tunnel Hearder格式为TCP报文，可以实现在网卡上的分片，而不使用CPU。


云计算：有一个平台，可以集中管理所有虚拟机资源，包括存储、计算节点、虚拟网络等。当有虚拟机变更(增删迁移)时，只需要将该虚拟机所需的资源参数输入，然后平台就可疑自动计算出所有需要的资源并自动分配和配置。即云计算平台，OpenStack/Cloudstack/AWS(Amozon)


SDN和网络虚拟化的关系：网络虚拟化平台或者云计算平台在网络控制部分就是集中化控制(NVP明确使用OpenFlow进行集中控制)，所以SDN controller可以集成到云计算平台中，极大程度地实现自动化。

第7章 SDN真实应用案例分析


Google的B4网络
1.背景
（1）Google的网络类型：数据中心内部网络/WAN网(数据中心之间互联的网络:G-Scale Network/面向Interet用户访问的网络:I-Scale Network)。改造的为G-Scale Network。
（2）数据类型：用户数据备份/扩数据中心存储访问(资源分布在不同的数据中心)/大规模的数据同步。数据量依次变大，对时延的敏感度依次降低，优先级依次降低。
（3）预期：分给数据同步的带宽越大越好，但可疑容忍偶尔的拥塞丢包、链路不通以及高时延/通过控制应用数据优先级和网络边缘的突发流量来优化路径/提高连接通道的带宽利用率。
2.实现

分为3层：物理设备层、局部网络控制层、全局控制层，一个site就是一个数据中心。流量从第一层交换机中传输第二层，经由RAP将消息传递给Quagga计算路由，然后在控制器中备份，下发到交换机中。TE Agent负责将链路信息传递给Gateway，在Gateway中维持全局拓扑，然后在TEServer中计算路由，下发到交换机中。
（1）网络设备层
交换机是由Google设计的，运行OpenFlow协议，使用TTP方式(使用传统网络的芯片实现SDN的功能)。每个数据中心中有多个交换机。
（2）局部网络控制层
RAP：Routing Application Proxy，负责将消息传递给Quagga。
Quagga：一个开源的路由协议栈，支持多种三层协议。B4中使用BGP和IS-IS。
Paxos：一个交换机连接多个Controller，但只有一个处于运行状态，即通过Paxos程序选举。基于功能选举，如控制功能A选举Controller1，控制功能B选举Controller2.
（3）全局控制层
TE生成的路由存储在ACL中，而Quagga计算的路由存在路由表中，会优先匹配ACL。从内部转发到别的数据中心的数据，使用ACL流；处理从别的数据中心发到本地数据中心内部的数据，使用路由表中的流。


ADVA、IBM和Marist College：基于SDN的虚拟光传输网络
ADVA是德国一家光网络公司，传统的WDM(波分复用)很有名。该项目将SDN技术应用到物理层，使用光传输设备FSP 3000互联3个数据中心。


Hybrid交换机包括两种：
一种在原有交换机的基础上将OpenFlow作为一个功能加入进去，使用port/Vlan隔离；一种是在OpenFlow交换机上融入传统二三层功能，类似OpenFlow中的“Go To Normal”指令，使用先ACL表匹配流规则，没有匹配成功再有二层转发处理。
